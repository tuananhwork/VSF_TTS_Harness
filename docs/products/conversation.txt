# Phương án: Tự động đúc rút Skill từ quá trình sử dụng Claude Cowork
**Mục tiêu:** Nhân viên dùng Cowork một thời gian → hệ thống tự phân tích lại các phiên làm việc → đúc rút quy trình lặp lại thành Skill → đưa vào sử dụng chung cho giai đoạn tiếp theo.
**Ngày soạn:** 12/06/2026
---
## 1. Đánh giá lại các giả định ban đầu
Trước khi chốt kiến trúc, cần điều chỉnh một vài giả định trong hướng xử lý ban đầu, vì hệ sinh thái Claude đã thay đổi nhanh trong 6 tháng qua:
**Giả định "việc tạo skill hiện phải làm thủ công, chưa tự động được" — đã lỗi thời một phần.** Hiện đã có các meta-skill chính thức và cộng đồng làm việc này ở mức bán tự động:
- `skill-creator` (Anthropic) — skill chuyên tạo/sửa/đánh giá skill khác, có cả benchmark và tối ưu mô tả trigger.
- `/run-skill-generator` — skill bundled trong Claude Code, tự ghi lại "công thức" chạy một quy trình từ môi trường sạch rồi commit thành skill cho project.
- Các skill cộng đồng kiểu `generating-skills-from-logs` — phân tích session history + CLI history rồi tự đề xuất và sinh ra skill hoàn chỉnh.
Cái **chưa có sẵn** là: khai thác chéo nhiều người dùng (org-wide), chấm điểm pattern nào đáng đúc thành skill, và vòng quản trị (review → publish → phân phối). Đây mới là phần ta cần tự xây — và nó nhẹ hơn nhiều so với "xây một AI tự học từ đầu".
**Giả định "cần knowledge graph index" — nên hoãn (YAGNI).** claude-mem đã có sẵn SQLite (metadata có cấu trúc) + vector store (Chroma) cho semantic search. Với mục tiêu "tìm pattern lặp lại trong các phiên", search ngữ nghĩa + group theo metadata (user, project, tool sequence) là đủ cho v1. Knowledge graph chỉ đáng đầu tư khi cần liên kết thực thể chéo hệ thống (ví dụ: cùng một quy trình "đối chiếu hóa đơn" xuất hiện dưới nhiều tên gọi ở VP và VHM). Đưa vào Phase 3, không đưa vào MVP.
**Giả định "trigger bằng scheduler của Claude Desktop" — đúng nhưng có giới hạn cần biết.** Cowork đã có scheduled tasks (`/schedule`, từ 25/02/2026), tuy nhiên **chỉ chạy khi máy của người dùng đang thức và Claude Desktop đang mở**. Với meta-agent chạy ở cấp tổ chức, nên cân nhắc chạy tập trung trên một máy chủ/máy dedicated bằng Claude Code headless (`claude -p` + cron) hoặc cloud scheduled tasks của Claude Code, thay vì phụ thuộc laptop nhân viên.
**Điểm rủi ro kỹ thuật cần intern xác minh sớm:** claude-mem là **plugin của Claude Code**, hoạt động dựa trên 5 lifecycle hooks (SessionStart, UserPromptSubmit, PostToolUse, Stop, SessionEnd). Cowork hỗ trợ plugin (bundle skills + connectors + sub-agents), nhưng cần kiểm chứng xem hooks của claude-mem có chạy đầy đủ trong môi trường Cowork hay không. Nếu không, fallback là dùng chính transcript/session memory mà Cowork lưu cục bộ, hoặc dùng "wrap-up skill" (mô tả ở Phase 0).

## 2. Kiến trúc đề xuất: 3 tầng + 1 vòng quản trị
```
┌─────────────────────────────────────────────────────────────┐
│  TẦNG 1 — CAPTURE (trên máy mỗi nhân viên)                  │
│  claude-mem (hoặc wrap-up skill) ghi lại observations        │
│  → SQLite + vector store cục bộ                              │
│  → Scheduled task hằng ngày: xuất session summaries          │
│    (đã lọc dữ liệu nhạy cảm) → push lên repo/share chung     │
└──────────────────────────┬──────────────────────────────────┘
                           ▼
┌─────────────────────────────────────────────────────────────┐
│  TẦNG 2 — CONSOLIDATION (kho tập trung)                     │
│  Git repo / SharePoint: session summaries theo user/tuần     │
│  Index: SQLite + embeddings (KG: hoãn đến Phase 3)           │
└──────────────────────────┬──────────────────────────────────┘
                           ▼
┌─────────────────────────────────────────────────────────────┐
│  TẦNG 3 — META-AGENT ("dream cycle", chạy hàng tuần)        │
│  Claude Code headless / Cowork scheduled task trên máy chủ   │
│  1. Mine pattern: quy trình lặp ≥ N lần, chuỗi tool ổn định  │
│  2. Chấm điểm ứng viên (frequency × effort saved × stability)│
│  3. Dùng skill-creator để draft SKILL.md + scripts           │
│  4. Đẩy vào REVIEW QUEUE (pull request)                      │
└──────────────────────────┬──────────────────────────────────┘
                           ▼
┌─────────────────────────────────────────────────────────────┐
│  VÒNG QUẢN TRỊ (human-in-the-loop)                          │
│  Skill owner review PR → merge → publish plugin marketplace  │
│  → nhân viên nhận skill mới qua plugin update                │
└─────────────────────────────────────────────────────────────┘
```
### Tầng 1 — Capture
Hai lựa chọn, có thể chạy song song:
**Lựa chọn A — claude-mem (tự động, giàu dữ liệu).** Hook `PostToolUse` ghi lại mọi tool call (file đọc, lệnh chạy, lỗi gặp), background worker nén thành observations bằng AI, lưu SQLite + vector DB cục bộ. Ưu điểm: không cần nhân viên làm gì, dữ liệu chi tiết đến mức chuỗi thao tác. Nhược điểm: cần xác minh tương thích Cowork; dữ liệu thô có thể chứa thông tin nhạy cảm.
**Lựa chọn B — Wrap-up skill (chủ động, nhẹ, chắc chắn chạy được trong Cowork).** Một skill nội bộ `session-wrapup`: cuối mỗi phiên làm việc đáng kể, nhân viên gõ `/wrapup` (hoặc skill tự trigger khi phát hiện task hoàn tất), Claude tự tóm tắt: mục tiêu phiên, các bước đã làm, tool/connector đã dùng, chỗ vướng và cách xử lý, đánh giá "quy trình này có lặp lại không". Output là một file markdown chuẩn hóa (schema cố định) lưu vào thư mục đồng bộ. Ưu điểm: triển khai được ngay tuần này, kiểm soát được nội dung xuất ra. Nhược điểm: phụ thuộc kỷ luật người dùng (giảm thiểu bằng cách nhúng vào quy trình/KPI thử nghiệm).
Khuyến nghị: **bắt đầu bằng B, pilot A song song trên nhóm nhỏ** (5–10 người dùng Claude Code/Cowork thành thạo).
### Tầng 2 — Consolidation
- Mỗi máy có một scheduled task hằng ngày: gom summaries mới → chạy bước **lọc dữ liệu nhạy cảm** (PII, số liệu tài chính, tên khách hàng — chính bước này cũng là một skill) → push lên git repo trung tâm (hoặc SharePoint, tận dụng hạ tầng Graph API sẵn có của V-APP 2).
- Repo trung tâm tổ chức theo `summaries/{username}/{yyyy-ww}/`.
- Index tập trung: một job đơn giản nạp summaries vào SQLite + embeddings để meta-agent search. Không cần KG ở giai đoạn này.
### Tầng 3 — Meta-agent (chu trình "dream")
Ý tưởng "dream/Hermes" về bản chất là **sleep-time consolidation**: agent chạy offline, đọc lại trải nghiệm, đúc rút thành tri thức tái sử dụng. Không cần framework đặc thù — một phiên Claude Code headless chạy theo lịch với prompt + skill phù hợp là đủ:
**Lịch chạy:** hàng tuần (cuối tuần), trên máy chủ dedicated. Dùng cron + `claude -p` hoặc cloud scheduled tasks; tránh phụ thuộc scheduler của Claude Desktop trên máy cá nhân vì giới hạn "máy phải thức, app phải mở".
**Pipeline của một chu trình:**
1. **Mine** — search index tìm các quy trình lặp: cùng mục tiêu xuất hiện ở ≥ 3 phiên hoặc ≥ 2 người dùng; chuỗi tool/bước ổn định; có "bài học sửa lỗi" (chỗ Claude làm sai rồi được người dùng chỉnh — đây là tri thức giá trị nhất để đưa vào skill).
2. **Score** — chấm điểm ứng viên theo công thức đơn giản: `tần suất × thời gian tiết kiệm ước tính × độ ổn định của quy trình − độ rủi ro (dữ liệu nhạy cảm, thao tác ghi/xóa)`. Chỉ những ứng viên trên ngưỡng mới đi tiếp.
3. **Draft** — gọi `skill-creator` để sinh `SKILL.md` (mô tả trigger song ngữ Việt–Anh như các skill nội bộ hiện có, ví dụ pattern của `vingroup-leave`), kèm scripts/templates nếu cần, kèm 3–5 test case ("golden questions") để đánh giá skill.
4. **Self-eval** — chạy thử skill draft trên test case (skill-creator hỗ trợ eval), ghi kết quả vào PR description.
5. **Submit** — mở pull request vào repo `cowork-skills` của công ty. **Meta-agent không bao giờ tự publish.**
**Ngoài tạo mới, meta-agent còn làm 2 việc:** (a) đề xuất **cải tiến skill hiện có** khi thấy người dùng liên tục phải sửa output của một skill; (b) đề xuất **gộp/loại bỏ** skill trùng lặp hoặc không ai dùng (đo qua tần suất xuất hiện trong summaries).
### Vòng quản trị (bắt buộc, không bỏ)
Skill về bản chất là **instruction injection** vào agent — skill kém chất lượng hoặc bị nhồi nội dung độc hại sẽ ảnh hưởng mọi người dùng nó. Do đó:
- Mỗi domain (kế toán, HR, IT…) có 1 **skill owner** review PR: kiểm tra tính đúng nghiệp vụ, mô tả trigger không đụng độ skill khác, không chứa dữ liệu nhạy cảm, thao tác ghi/xóa có bước confirm.
- Merge → CI đóng gói thành **plugin** nội bộ (plugin = bundle skills + connectors + sub-agents) → publish lên marketplace nội bộ → nhân viên cập nhật qua plugin update. Versioning theo semver trong frontmatter của SKILL.md.
- Theo dõi sau phát hành: meta-agent tuần sau đọc summaries để xem skill mới có được trigger đúng không, có bị nhân viên "sửa tay" output không → vòng lặp tự tối ưu.

## 3. Lộ trình triển khai
**Phase 0 — Thủ công có cấu trúc (tuần 1–2).** Tạo `session-wrapup` skill + repo `cowork-skills` + quy trình PR review. Nhân viên tự đề cử quy trình đáng đúc thành skill; dùng `skill-creator` tạo thủ công 3–5 skill đầu tiên. Mục đích: dựng xương sống quản trị trước khi tự động hóa, và có ngay giá trị để báo cáo lãnh đạo.
**Phase 1 — Capture tự động (tuần 2–4).** Intern pilot claude-mem trên nhóm nhỏ: xác minh tương thích Cowork (nếu không tương thích → đánh giá session memory/transcript có sẵn của Cowork làm nguồn thay thế); dựng scheduled task xuất + lọc + push summaries; dựng index tập trung. Deliverable của intern: báo cáo tương thích + script export chuẩn hóa.
**Phase 2 — Meta-agent (tuần 4–8).** Dựng máy chủ chạy chu trình tuần; viết prompt mining + công thức scoring; nối với skill-creator và quy trình PR. Chạy song song với đề cử thủ công 1 tháng, so sánh chất lượng skill do meta-agent đề xuất với skill do người đề cử.
**Phase 3 — Mở rộng (khi có nhu cầu thực).** Knowledge graph index nếu cần liên kết quy trình chéo entity VP/VHM; dashboard đo lường (skill nào dùng nhiều, tiết kiệm bao nhiêu); mở rộng nguồn capture sang Claude Code của đội kỹ thuật.
